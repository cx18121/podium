<task type="auto">
  <name>Task 1: Fix stop-sequence race condition in useRecording.ts</name>
  <files>src/hooks/useRecording.ts</files>
  <action>
In `recorder.onstop` (around line 103), the current call order is:

```
stopTimer();
stopStream();           // <-- calls mlWorker.cleanupWorker() which sets workerRef.current = null
const visualEvents = await mlWorker.stopWorker();  // <-- workerRef is null, resolves []
```

`stopStream()` is defined as:
```ts
const stopStream = useCallback(() => {
  streamRef.current?.getTracks().forEach((t) => t.stop());
  streamRef.current = null;
  mlWorker.cleanupWorker();  // terminates worker
}, [mlWorker]);
```

Fix: call `mlWorker.stopWorker()` BEFORE `stopStream()`. The worker must still be alive when we flush its `pendingEvents`.

New order inside `recorder.onstop`:

```ts
recorder.onstop = async () => {
  setStatus('processing');
  stopTimer();

  const durationMs = Date.now() - startTimeRef.current;
  const rawBlob = new Blob(chunksRef.current, { type: 'video/webm' });

  // Flush ML worker events BEFORE stopping the stream — worker must be alive to respond
  const visualEvents = await mlWorker.stopWorker();

  // Now stop the stream (also calls cleanupWorker internally)
  stopStream();

  const fixedBlob = await webmFixDuration(rawBlob, durationMs);

  onRecordingReady({
    fixedBlob,
    durationMs,
    autoTitle: formatAutoTitle(),
    visualEvents,
  });
};
```

Note: `stopStream()` calls `cleanupWorker()` which terminates the worker after a 500ms grace period. That is fine — `stopWorker()` has already received the `events` message before `cleanupWorker()` is called.

Do NOT change any other logic (timer, blob assembly, webmFixDuration, onRecordingReady).
